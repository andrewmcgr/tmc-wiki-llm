# Agent Write Protocol

**Type:** template
**Summary:** How AI agents should create and modify files to maintain standard compliance.
**Tags:** #meta #automation
**Status:** active
**Updated:** 2026-04-07

---

## Purpose

This protocol ensures that all agent-written files are:
1. Discoverable by future queries (standard metadata)
2. Auditable (source attribution)
3. Human-reviewable before becoming operational
4. Consistent across sessions

---

## The Golden Rule: Inbox First

When capturing something new, it goes to `_inbox/` first — not directly to a final location. This keeps agent writes auditable and gives the user the chance to review before content becomes part of the system.

**Exceptions — agents MAY write directly to final locations:**
- `WIKI-LOG.md` — append-only evolution log
- Files the user explicitly says to create in a specific location

Everything else → `_inbox/` first, with an offer to file immediately.

---

## Required: Metadata Block

Every file an agent creates or significantly modifies must include the metadata block. See `_config/standard.md` for the full spec.

**Minimum required fields for agent-written files:**
```markdown
**Type:** [knowledge | decision | initiative | meeting | ...]
**Summary:** [One sentence — must be informative, not generic]
**Tags:** #[domain] #[type]
**Status:** draft
**Updated:** YYYY-MM-DD
**Source:** auto-generated by [agent-name]
```

The `**Source:**` field identifies the originating agent and allows filtering during triage.

---

## Inbox File Naming

Agent-created inbox files must follow:
```
_inbox/YYYY-MM-DD-[short-description].md
```

Examples:
- `_inbox/2026-04-07-deployment-process-capture.md`
- `_inbox/2026-04-08-meeting-notes-standup.md`

---

## Updating Existing Files

When modifying an existing file:

1. **Update the `Updated:` field** to today's date
2. **Update the `Summary:` line** if the current state changed meaningfully
3. **Append, don't overwrite** — prefer appending to existing content unless the file schema requires replacement

---

## Status Convention

Agent-written files always start with `Status: draft` unless updating a file with an established status. This signals that the content needs human review.

---

## Source Preservation

When filing an inbox item to its final location, check `_config/config.md` for the `Preserve Important Sources` setting.

**If set to `yes`:** Before discarding the original inbox file, assess whether the source is worth preserving:
- Is this a document the user may want to re-process with a better model later?
- Is this a clipped article whose URL might disappear?
- Is this a key reference document (architecture, spec, research paper)?

If yes, move the original to `_sources/YYYY-MM-DD-description.md` instead of discarding it. The filed wiki version in its final location is the working copy; the source is the immutable original.

**If set to `no` (default):** Discard inbox items normally after filing. No source preservation.

During the training phase, ask the user once or twice to calibrate: "I kept the original of [X] in `_sources/` because it seems important. Want me to do this for significant sources going forward?"

---

## What Agents Must NOT Do

- Write directly to `4-Private/` — sensitive content requires human judgment
- Delete or overwrite existing files without explicit instruction
- Create files without the metadata block
- Write files with a generic Summary like "Auto-generated file" — must be informative
- Create duplicate files when one already exists on the topic — update the existing one instead
